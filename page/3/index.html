Getter与Setter是用来获得或者设置变量值得函数或者方法
Getter与Setter的概念是在计算机编程中一个普遍的概念
几乎所有高级编程语言
几乎所有的高级编程语言都带有一组用于实现getter和setter的语法，包括JavaScipt

在这篇文章 我们将能懂getters setters是什么 如何建立和使用他们在Javascript

Getters-setters与封装

Getters-setters总是会与封装联系在一起
封装可以用两种方式来理解

  1. 首先 它是数据获取者、设置者和修改者的集合 这种限定在某些操作的时候是非常有意义的
    例如验证 在数据保存和展示之前 getter和setter提供了完美解决

  2. 依据封装是为了隐藏数据 使得外部不能访问内部数据 除了通过getter和setter 这样避免了程序中其他代码重写数据

创建Getters-setters

  1. 通过方法

由于getter和setter本质改变和设置值的函数 

var obj = {
    foo:    'this is the value of foo',
    getFoo: function() {
                return this.foo;
    ,
    setFoo: function(val) {
                this.foo = val;
    }
}

    console.log(obj.getFoo());
    // "this is the value of foo"

    obj.setFoo('hello');

    console.log(obj.getFoo());
    // "hello"
    This is the simples

  1. 通过关键字

    通过关键字get/set是官方和健壮的做法

var obj = {
    fooVal: 'this is the value of foo',
    get foo() {
        return this.fooVal;
    },
    set foo(val) {
        this.fooVal = val;
    }
}

    console.log(obj.foo);
    // "this is the value of foo"

    obj.foo = 'hello';

    console.log(obj.foo);
    // "hello"

注意，数据只能存储在与getter-setter方法不同的属性名(fooVal)下，因为持有getter-setter方法的属性也不能持有数据。

Overwrite prevention 重写保护

如果因为某种原因选择第一种 想使属性通过getter-setter方法并且是一个只读属性
可以通过Object.defineProperties
Object.defineProperties, Object.defineProperty and Reflect.defineProperty automatically configure to writable: false which means read-only:

/* Overwrite prevention */
var obj = {
foo: 'this is the value of foo'
};

Object.defineProperties(obj, {
'getFoo': {
    value: function () {
        return this.foo;
    }
},
'setFoo': {
    value: function (val) {
        this.foo = val;
    }
}
});

obj.getFoo = 66;
// getFoo is not going to be overwritten!

console.log(obj.getFoo());
// "this is the value of foo"

what-is-closure?

Posted on 2019-11-22

closure 闭包

什么是封闭？闭包是捆绑在一起（封闭）的函数及其周围状态（词汇环境）的组合。
换句话说，闭包使您可以从内部函数访问外部函数的范围。
在JavaScript中，每次创建函数时都会在函数创建时创建闭包。
要使用闭包，请在另一个函数中定义一个函数并将其公开。
要公开一个函数，请将其返回或传递给另一个函数。
即使返回了外部函数，内部函数也可以访问外部函数范围内的变量。

A closure is the combination of a function bundled together (enclosed) with references to its surrounding state (the lexical environment).
In other words, a closure gives you access to an outer function’s scope from an inner function.
In JavaScript, closures are created every time a function is created, at function creation time.
To use a closure, define a function inside another function and expose it. To expose a function, return it or pass it to another function.
The inner function will have access to the variables in the outer function scope, even after the outer function has returned.

在JavaScript中，闭包是用于实现数据隐私的主要机制。

当使用闭包进行数据保密时，包含的变量仅在包含（外部）函数的范围内。

除了通过对象的特权方法之外，您无法从外部范围获取数据。

在JavaScript中，闭包范围内定义的任何公开方法都具有特权。

In JavaScript, closures are the primary mechanism used to enable data privacy.

When you use closures for data privacy, the enclosed variables are only in scope within the containing (outer) function.

You can’t get at the data from an outside scope except through the object’s privileged methods.

In JavaScript, any exposed method defined within the closure scope is privileged. 

const getSecret = (secret) => {
    return {
        get: () => secret
    };
    };

    const msg = '.get() should have access to the closure.';

    } catch (e) {
        assert.ok(true, `The secret var is only available

    }

    assert.equal(actual, expected, msg);
    assert.end();
});

2 函数式编程

  • 函数式编程通过组合数学函数来生成程序，并避免共享状态和可变数据
  • Lisp（在1958年指定）是最早支持函数式编程的语言之一，并且受到lambda演算的极大启发。
  • Lisp和许多Lisp家庭语言在今天仍然很普遍

  1. 纯函数/函数纯度。

  2. 避免副作用。

  3. 简单的功能组成。

  4. 功能语言的示例：Lisp，ML，Haskell，Erlang，Clojure，Elm，F Sharp，OCaml等

  5. 提到支持FP的功能：一流的功能，高阶功能，作为参数/值的功能

3.经典继承和原型继承有什么区别？

  1. 类继承：实例从类继承（如蓝图-类的描述），并创建子类关系：分层类分类法。

    实例通常通过带有`new`关键字的构造函数来实例化。
类继承可以使用也可以不使用ES6中的`class`关键字。

  2. 原型继承：实例直接从其他对象继承。

    通常通过工厂函数或Object.create（）实例化实例。
实例可以由许多不同的对象组成，从而可以轻松地进行选择性继承。

  3. 类：创建紧密耦合 或层次结构/分类法。 原型：提及串联继承，原型委托，功能继承，对象组成。

4. 函数式程序设计与面向对象程序设计的优缺点是什么？

  • OOP优点：易于理解对象的基本概念，并易于解释方法调用的含义。OOP倾向于使用命令式而不是声明式，其读起来就像一组简单的指令供计算机遵循。

  • OOP缺点：OOP通常取决于共享状态。对象和行为通常在同一实体上固定在一起，可以由具有不确定顺序的任意数量的函数随机访问，这可能导致不良行为，例如竞态条件。

  • FP优点：使用功能范式，程序员避免了任何共享状态或副作用，从而消除了由多个功能争夺同一资源引起的错误。

    通过使用无点样式（又称为默认编程）等功能，与OOP相比，对于那些更普遍可重用的代码，功能往往会得到根本简化，并且易于重组。

    FP还倾向于使用声明性和指示性样式，它们不会详细说明操作的分步说明，而是专注于执行操作，让底层函数负责操作方式。

    这为重构和性能优化留出了极大的自由度，甚至允许您通过很少的代码更改就可以用效率更高的算法替换整个算法。例如，记住或使用懒惰的评估代替急切的评估

    利用纯函数的计算也很容易在多个处理器或分布式计算集群之间扩展，而不必担心线程资源冲突，竞争条件等。

  • FP缺点：过度利用FP功能（如无点样式和较大的组合）可能会降低可读性，因为生成的代码通常被更抽象地指定，更简洁且不太具体。

    比函数编程更熟悉OO和命令式编程，因此即使是函数式编程中的常见习语也会使新团队成员感到困惑。FP的学习曲线比OOP陡得多

    这是因为OOP的广泛普及使OOP的语言和学习材料变得更具对话性，而FP的语言往往更具学术性和形式性。

    FP概念经常被编写为使用lambda演算，代数和范畴论中的惯用语和记号，所有这些都需要先了解这些领域的知识基础。

  • 提到共享状态带来的麻烦，争夺相同资源的不同事物，等等……

  • 意识到FP能够从根本上简化许多应用程序的能力

  • 意识到学习曲线的差异

  • 阐明副作用及其对程序可维护性的影响

  • 意识到功能强大的代码库可能具有陡峭的学习曲线

  • 意识到，与同等的FP代码库相比，高度面向对象的代码库可以极大地抵抗更改并且非常脆弱

  • 意识到不变性会带来极其可访问和可延展的程序状态历史记录，从而允许轻松添加无限的撤消/重做，倒带/重播，时间旅行调试等功能

  • 在这两种范例中都可以实现不变性，但是共享状态对象的泛滥使OOP的实现变得复杂

7.“偏向于类继承而不是对象继承”是什么意思？

这摘自“设计模式：可重用的面向对象软件的元素”。这意味着应该通过将较小的功能单元组装到新对象中，而不是从类继承并创建对象分类法来实现代码重用

换句话说，使用can-do，has-a或uses-a关系代替is-a关系。
  • 避免类层次结构。
  • 避免脆性基类问题。
  • 避免紧密耦合。
  • 避免使用严格的分类法（强迫is-a关系对于新用例最终是错误的）。
  • 避免大猩猩香蕉问题（“您想要的是香蕉，得到的是拿着香蕉的大猩猩，以及整个丛林”）。
  • 使代码更灵活。

8.什么是双向数据绑定和单向数据流，它们有何不同？

  • 双向数据绑定意味着将UI字段动态绑定到模型数据，这样，当UI字段更改时，模型数据也会随之更改，反之亦然。
  • 单向数据流的一种方式意味着模型是事实的唯一来源。UI中的更改会触发消息，向用户发送信号通知模型（或在React中store）的意图。只有模型才有权更改应用程序的状态。结果是数据始终沿单一方向流动，这使它更易于理解。

  • 单向数据流是确定性的一种方式，而双向绑定可能会引起副作用，而这些副作用很难理解和理解。

  • React是单向数据流的新典范示例，因此提到React是一个很好的信号。

  • Cycle.js是另一种流行的单向数据流实现。
  • Angular是使用双向绑定的流行框架

9. What are the pros and cons of monolithic vs microservice architectures?

  • 整体架构意味着您的应用程序是作为一个内聚的代码单元编写的，其组件被设计为可以协同工作，共享相同的内存空间和资源

  • 微服务架构意味着您的应用程序由许多较小的独立应用程序组成，这些应用程序可以在自己的内存空间中运行，并且可以在可能的许多不同机器上彼此独立地扩展

  • 整体式优点：整体式架构的主要优势在于，大多数应用程序通常具有大量的跨领域问题，例如日志记录，速率限制以及诸如审计跟踪和DOS保护之类的安全功能。
    当所有事情都通过同一个应用程序运行时，很容易将组件与那些跨领域的问题联系起来。由于共享内存访问比进程间通信（IPC）更快，因此也可能具有性能优势。

  • 整体式劣势：随着应用程序的发展，整体式应用程序服务趋于紧密耦合和纠缠，从而难以出于独立缩放或代码可维护性等目的隔离服务。
    整体架构也很难理解，因为当您查看特定的服务或控制器时，可能存在不明显的依赖关系，副作用和魔力。

  • 微服务的优点：微服务架构通常组织得更好，因为每个微服务都有非常具体的工作，并且与其他组件的工作无关。
    解耦的服务也更易于重组和重新配置，以服务于不同应用程序的目的（例如，服务于Web客户端和公共API）
    根据它们的组织方式，它们还可以具有性能优势，因为可以隔离热服务并独立于应用程序的其余部分进行扩展

  • 微服务的弊端：在构建新的微服务架构时，您可能会发现很多跨领域的问题，而这些问题在设计时并未预料到。
    单一的应用程序可以建立共享的魔术助手或中间件来处理此类跨领域的问题，而无需付出太多努力。
    在微服务架构中，您要么需要为每个横切关注点承担单独模块的开销，要么将横切关注点封装在所有流量都通过的另一个服务层中。
    最终，即使是单一的体系结构也倾向于通过外部服务层路由流量，以解决跨领域的问题，
    但是采用整体体系结构，有可能将这项工作的成本推迟到项目更加成熟之前。
    微服务经常部署在自己的虚拟机或容器上，从而导致VM争用工作激增。
    这些任务通常使用集装箱车队管理工具自动执行。

    • 尽管与单一应用程序相比，初始成本较高，但对微服务仍持积极态度。
    • 意识到微服务从长远来看往往具有更好的性能和扩展性。关于微服务与整体应用程序的实践。
    • 对应用程序进行结构设计，以使服务在代码级别上彼此独立，但一开始就很容易将其作为整体应用程序捆绑在一起。
    • 微服务的间接成本可以延迟，直到支付价格变得更加实际为止。

    10.什么是异步编程，为什么它在JavaScript中很重要？

    同步编程意味着，除非有条件和函数调用，否则代码将自上而下依次执行，从而阻止长时间运行的任务（例如网络请求和磁盘I / O）。
异步编程意味着引擎在事件循环中运行。当需要阻止操作时，请求将启动，并且代码将继续运行而不会阻止结果。当响应准备就绪时，将触发一个中断，这将导致事件处理程序运行，并在此处继续控制流程。
这样，单个程序线程可以处理许多并发操作。用户界面本质上是异步的，它们花费大部分时间等待用户输入来中断事件循环并触发事件处理程序。
默认情况下，Node是异步的，这意味着服务器的工作方式几乎相同，在循环中等待网络请求，并在处理第一个请求时接受更多的传入请求。
这在JavaScript中很重要，因为它非常适合用户界面代码，并且对服务器性能非常有益。
